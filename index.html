在软件设计领域，设计模式是对通用问题的可复用的解决方案。设计模式是一系列帮你写出更可理解和复用代码的模板，设计模式帮你创建松耦合的代码以便你不需要费多大力就可以改变或者替换代码中的组件。

如果你对设计模式还很陌生，那么告诉你一个好消息！在 iOS 的开发过程中，其实你不知不觉已经用了很多设计模式。这得益于 Cocoa 提供的框架和一些良好的编程习惯。接下来的这篇教程将会带你一起飞，去领略设计模式的魅力。

  • 设计模式是什么？

MVC（模型-视图-控制器）模式-设计模式之王


模型-视图-控制器（MVC）是Cocoa的构建基础，也是应用最广使用最频繁的设计模式。MVC 根据类的大体功能将他们分为三个功能模块，并鼓励将不同功能的代码写到对应的功能模块中，从而使得三个功能模块之间具有一个清晰简洁的分工。

作为划分依据的三个基本职责是：

  • Model（模型）:
    存储数据并且定义如何操作这些数据。在我们的例子中，就是 Album 类。

  • View（视图）:
    负责模型层的可视化展示，并且负责用户的交互，一般来说，View 所指的是 UIView 以及所有继承自 UIView 的类。在我们的项目中就是 AlbumView 这个类。

  • Controller（控制器）:
    控制器是用于充当上面两者的中介，协调二者之间所有的交互工作。负责访问Model中的数据并在 View 中将他们展示出来，还负责事件的监听并按需操作 Model中的数据。能猜到哪个是我们的视图控制器么？没错，就是ViewController

一个MVC设计模式的良好实践意味着每一个对象都被划入到这三个模块中的一个。

下图很好的展示了 View 和 Model 通过 Controller 进行交互的过程：


Model会把任何的数据变更信息告知Controller，然后由Controller对View上面显示的内容进行更新。反之View 会把用户操作信息反馈给 Controller，然后 由Controller来按需更新Model，或者在 Model 中检索被请求的数据。

你可能会想，既然这样，那为什么不直接搞一个 Controller，然后把View 和 Model 都写在里面，这样岂不简单的多？

之所以使用 MVC 设计模式的最大原因在于使代码分离（解耦）以及提高代码的复用性。理论上，View 应该与 Model 完全隔离。这样如果该 View 不依赖于某一 Model的特定实现，它便可以复用于不同的 Model。

举个例子来说，如果将来你打算把电影或者书籍放到资料库中，你仍然可以使用这个AlbumView来展示它们。再者，如果你以后想创建一个与专辑有关的新工程，你可以直接把Album类的两个文件拽过去用，因为它不依赖与任何View。这就是MVC的强大之处。

  • 如何使用MVC模式

    首先，你需要确保你工程中的每个类都被划分到 Model、View、Controller 相对应的模块中；不要在某一个类中组合两个模块的功能。到目前为止，你已经创建了一个Album类和AlbumView类，这样的命名方式本身就融入了 MVC 的理念。

    其次，为了确保将来创建出的类也都符合这个分工思想，你应该创建三个文件组并将代码源文件分别存放到对应的文件组中。

    在 Xcode 菜单中点击File\New\Group（或者按下Command + Option + N，命名组为Model，重复同样的过程来创建View和Controller组。

    现在将Album.h和Album.m移到Model组，将AlbumView.h和AlbumView.m移到View组，最后将ViewController.h和ViewController.m移动到Controller 组。

Singleton Pattern（单例）模式

单例设计模式确保指定类只存在一个实例，并且该实例留有一个全局访问接口。单例模式通常采用懒加载的方式创建实例，即该实例将会在第一次被访问时创建。

注：苹果大量使用了此模式。例如：[NSUserDefaults standardUserDefaults]，[UIApplication sharedApplication]，[UIScreen mainScreen]，[NSFileManager defaultManager]，所有的这些方法都返回一个单例对象

你很可能在想，为什么这么关心一个类是否存在有多个实例？创建一个实例又用不了多少代码和内存

有时候，某一特定类只拥有一个实例是必要的。举例来说，你不会用到多个Logger的实例，除非你需要同时往多个文件中写入日志。再比如，一个全局的配置文件处理类在对一个配置文件执行修改操作时，使用单一实例对文件进行修改要比同时使用多个更便于掌控线程的安全

  • 如何使用单例模式

    首先来看看下面的图：

    上图展示了一个单例类，该单例类有一个属性用于存储类的单一实例，以及sharedInstance和init两个方法。在sharedInstance消息第一次被发送时，instance属性尚未被初始化，这时候我们将该类的实例在类方法的内部初始化，然后返回实例的引用。

    当你下一次调用sharedInstance的时候，instance已再执行初始化操作，直接返回实例引用。这个逻辑得以确保该类始终只有这一个实例。

Facade Design Pattern 外观模式


外观模式在复杂的业务系统上提供了简单的接口。如果直接把业务的所有接口直接暴露给使用者，使用者需要单独面对这一大堆复杂的接口，学习成本很高，而且存在误用的隐患。如果使用外观模式，我们只要暴露必要的 API 就可以了。

下图演示了外观模式的基本概念：


API 的使用者完全不知道这内部的业务逻辑有多么复杂。当我们有大量的类并且它们使用起来很复杂而且也很难理解的时候，外观模式是一个十分理想的选择。

外观模式把使用和背后的实现逻辑成功解耦，同时也降低了外部代码对内部工作的依赖程度。如果底层的类发生了改变，外观的接口并不需要做修改。

举个例子，如果有一天你想换掉所有的后台服务，你只需要修改 API 内部的代码，外部调用 API 的代码并不会有改动。

  • 如何使用外观模式

现在我们用 PersistencyManager 来管理专辑数据，用 HTTPClient 来处理网络请求，项目中的其他类不应该知道这个逻辑。他们只需要知道 LibraryAPI 这个“外观”就可以了。

为了实现外观模式，应该只让 LibraryAPI 持有 PersistencyManager 和 HTTPClient 的实例，然后 LibraryAPI 暴露一个简单的接口给其他类来访问，这样外部的访问类不需要知道内部的业务具体是怎样的，也不用知道你是通过 PersistencyManager 还是 HTTPClient 获取到数据的。

注：单例的存在通常会伴随着应用的整个运行周期，所以在单例的内部不要设有过多的外部强引用指针，否则被引用对象在应用程序退出前不会被释放

大致的设计是这样的：

LibraryAPI 会暴露给其他代码访问，但是 PersistencyManager 和 HTTPClient 则是不对外开放的。

打开 LibraryAPI.h 导入以下头文件：

#import "Album.h"

接着，添加以下方法声明:

- (NSArray *)getAlbums;
- (void)addAlbum:(Album *)album atIndex:(int)index;
- (void)deleteAlbumAtIndex:(int)index;

目前只把这几个方法暴露给外界访问
来到，LibraryAPI.m 然后导入两个头文件

#import "PersistencyManager.h"
#import "HTTPClient.h"

这两个头文件将不会在任何其他地方被引入。再次强调：你的 API 是这一整个复杂体系的唯一接入点

现在我们到延展处加入一些私有实例变量（在 @implementation 行前）：

@interface LibraryAPI () {
    PersistencyManager *persistencyManager;
    HTTPClient *httpClient;
    BOOL isOnline;
}
 
@end

布尔值 isOnline用来标识当前是否为联网状态，如果是联网状态就会去网络获取数据。
我们需要在 init 里面初始化这些变量：

- (id)init {
    if (self = [super init]) {
        persistencyManager = [[PersistencyManager alloc] init];
        httpClient = [[HTTPClient alloc] init];
        isOnline = NO;
    }
    return self;
}

HTTPClient 并不会直接和真实的服务器交互，只是用来演示外观模式的使用。所以 inOnline 这个值我们一直设置为 NO

接下来在 LibraryAPI.m 里添加如下代码：

- (NSArray *)getAlbums {
    return [persistencyManager getAlbums];
}
 
- (void)addAlbum:(Album*)album atIndex:(int)index {
    [persistencyManager addAlbum:album atIndex:index];
    if (isOnline) {
        [httpClient postRequest:@"/api/addAlbum" body:[album description]];
    }
}
 
- (void)deleteAlbumAtIndex:(int)index {
    [persistencyManager deleteAlbumAtIndex:index];
    if (isOnline) {
        [httpClient postRequest:@"/api/deleteAlbum" body:[@(index) description]];
    }
}

看一下 addAlbum:atIndex: 这个方法，先更新本地缓存，然后如果是联网状态还需要向服务器发送网络请求。这便是外观模式的强大之处：如果外部文件想要添加一个新的专辑，它不会也不用去了解内部的实现逻辑是怎么样的。

注：当你设计外观的时候，请务必牢记：使用者随时可能直接访问你的隐藏类。永远不要假设使用者会遵循你当初的设计做事。

Decortor 装饰者模式

装饰者模式可以动态的给指定的类添加一些行为和职责，而不用对原代码进行任何修改。当你需要使用子类的时候，不妨考虑一下装饰者模式，可以在原始类上面封装一层

在 Objective-C 中，实现装饰者模式的方式有两种：分类(Category) 和代理(Delegation)

  • 分类

    分类是一种十分强大的机制，可以让你在不用继承的情况下，给已存在的类、结构体或者枚举类添加一些新的功能。分类中的代码会在运行时被加入到原有类中，并且可以像原有类的方法一样执行

    区别于经典装饰者模式的是分类中不持有原有类的实例变量

    要创建一个分类，在菜单中点击"File\New\File…"（或者按Command+N快捷键）选择 IOS > Source 但是这次不要习惯性的选中 Cocoa Touch Class，选中Objective-C File 然后点击 Next。在 File 输入框中填入 TableRepresentation，File Type 选框中选择 Category，Class 输入框中填入 Album 然后点击 Next > Create

    在 Album+TableRepresentation.h 中加入一个方法声明:

    - (NSDictionary *)tr_tableRepresentation;

    在方法的前面有个 tr_ 前缀，是 TableRepresentation 的缩写，这样有利于和类的原有方法进行区分，避免使用的时候产生冲突。现在很多还在维护中的第三方库都已经改成了这个风格。

    在 Album+TableRepresentation.m 中实现该方法:

    - (NSDictionary *)tr_tableRepresentation {
return @{@"titles":@[@"Artist", @"Album", @"Genre", @"Year"],
         @"values":@[self.artist, self.title, self.genre, self.year]};
}

    思考一下这个设计模式的强大之处：

    • 我们可以直接在扩展里使用 Album 里的属性

    • 我们给 Album 类添加了内容但是并没有继承它，也不会妨碍别的类去继承它

    • 这个简单的扩展让我们可以更好地把 Album 的数据展示在 UITableView 里，而且不用修改源码

    苹果在 Foundation 框架中大量使用到分类，你可以在 NSString.h 中，搜索@interface NSString，你会看到该类的定义部分以及它的四个分类: NSStringExtensionMethods，NSStringEncodingDetection，NSExtendedStringPropertyListParsing，NSStringDeprecated

    合理的使用分类可以增加代码的条理性，将不同的方法按照功能进行区块划分
